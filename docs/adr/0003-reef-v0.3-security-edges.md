# ADR 0003: adopt Reef v0.3 production security edges

- Status: accepted
- Date: 2026-07-18
- Decision owners: Coral maintainers

## Context

Coral used Reef v0.1 low-level TLS and bearer helpers. Those helpers could
materialize encryption/authentication but did not know the listener bind or
client target, so they could not reject external plaintext, bearer exposure,
redirect escape, or unsafe transport reuse. File-backed bearer tokens and
client trust material also lacked a Coral-owned observable lifecycle.

Reef v0.3.0 keeps those helpers for compatibility but defines `edge`,
`grpcreef.NewServerEdge`, `edge.NewHTTPServer`, and
`reefclient.NewEdgeTransport` as its production API.

The immutable v0.3.0 tag contains these APIs and their lifecycle tests, although
its changelog heading still says “Unreleased (target: v0.3.0)”. Coral pins the
tag checksum and treats that stale heading as an upstream documentation defect,
not as permission to consume Reef's untagged branch.

## Decision

Coral uses Reef v0.3.0 high-level edges for:

- OTLP gRPC and HTTP ingress;
- the self-observability HTTP listener;
- Amber and Fathom HTTP exporters for every signal.

External plaintext is fail-closed unless the configuration explicitly opts in.
Bearer over plaintext requires a second danger opt-in. HTTP exporters are
origin-bound and do not follow redirects.

All file-backed edge credentials use Reef managed last-known-good reload.
The owning ingress/exporter closes that lifecycle. Construction failure closes
already-materialized exporters in reverse order.

The Reef bearer principal is propagated unchanged through the OTLP request
context. It is authentication evidence, not yet a tenant ID. Coral will map it
to an immutable organisation/project only in the tenant-identity increment.

## Compatibility

TLS configurations and literal-loopback plaintext configurations remain valid.
Existing wildcard/external plaintext configurations must add `insecure: true`
or migrate to TLS. Bearer over plaintext must also add
`danger_allow_bearer_over_plaintext: true`.

These changes are intentionally visible because the former behavior could
expose telemetry or credentials. The complete migration is documented in
`docs/REEF_V0.3_MIGRATION.md`.

Standard OTLP payloads, Wisp optional headers, and Gyre v0.5.0 contracts do not
change. Legacy Jaeger/Zipkin receiver security is explicitly outside this
increment.

## Failure semantics

- invalid edge policy fails application construction before serving;
- initial credential load failure is fail-stop;
- reload failure retains last-known-good state and increments a bounded failure
  metric;
- successful rotation advances the observed generation;
- shutdown closes reload loops and idle exporter connections;
- redirects are returned as downstream responses rather than followed.

## Verification

Tests cover external plaintext rejection, double opt-in for bearer plaintext,
principal propagation, file-token rotation, lifecycle status/events,
origin/redirect containment, construction cleanup, race safety, and existing
OTLP transport behavior.
